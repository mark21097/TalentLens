Why do we create a talentlens folder?
Good question! The talentlens/ folder is your Python package — it's where all your reusable source code lives. Here's why it matters:

Separation of concerns: Notebooks (notebooks/) are for exploration and storytelling. The talentlens/ package holds the clean, tested, importable code that the notebooks (and later Streamlit) call.

Import anywhere: Because of the -e . in your requirements.txt (editable install), you can do from talentlens.config import POSTINGS_CSV from any notebook or script. No messy sys.path hacks.

Reusability: You write clean_postings() once in talentlens/dataset.py, then use it in notebooks, tests, and the Streamlit app without copy-pasting.

Professional standard: This is the Cookiecutter Data Science convention your README already describes. Hiring managers expect to see source code separated from notebooks.

The structure maps to your pipeline phases:

talentlens/config.py → all file paths and constants
talentlens/dataset.py → data loading and cleaning
talentlens/features.py → NLP feature engineering (Phase 2)
talentlens/modeling/ → ML training and prediction (Phase 3)
talentlens/rag/ → RAG pipeline with LangChain + FAISS (Phase 4)

"Would it be beneficial to map out this schema using an ER diagram and translate to SQL tables?"
Yes, absolutely. An ER diagram would be excellent here — especially since you learned this in Database Management. The schema is:


postings (PK: job_id) ──1:N── job_skills (FK: job_id, FK: skill_abr)
                       ──1:N── job_industries (FK: job_id, FK: industry_id)
                       ──1:N── benefits (FK: job_id)

companies (PK: company_id) ──1:N── company_industries

skills (PK: skill_abr)        ← lookup table
industries (PK: industry_id)  ← lookup table
A schema.py could define this formally, but for now documenting it in the notebook and data dictionary is sufficient. We can add a Draw.io ER diagram to the README later.

"So the secondary tables allow us to explore the data more in depth essentially?"
Exactly. The main postings table has the core info (title, description, salary). The secondary tables add dimensions — what skills a job requires, what industry it's in, what benefits are offered, how big the company is. These become features for ML later.

"Since our theme is on skills, we are now mapping the skills right?"

Notebook 02 Questions:

"I did all nrows, is it normal to use smaller samples?"
Very normal and professional. In industry, data scientists always prototype on a small sample first (10K-50K rows) to make sure the code works, then scale up to the full dataset. You save time debugging on a 2-second load instead of a 2-minute load. The nrows parameter is there specifically for this workflow.

"Are the critical columns all attributes or just the ones we're looking for?"
Just the ones relevant to our research themes. The raw table has ~27 columns, but we only profile the 12 that matter for our analysis (description for NLP, salary for Theme 1, days_open for Theme 2, applies/views for Theme 4, etc.). The others (like job_posting_url, application_url) aren't useful for modeling.

"We didn't get any duplicate job_id rows? Is this just good practice?"
Yes, it's defensive programming — you check even if you expect it to be clean. In real-world data pipelines, data arrives dirty in unpredictable ways. The fact that there were no dupes is good news, but the check costs nothing and would catch a real problem instantly. Always verify assumptions.

"The cleaning pipeline only cleaned 7 out of 123K rows"
That's actually a sign the dataset is high quality! Most Kaggle datasets are pre-curated. In production, you'd often drop 10-30% of rows. The 7 rows were likely missing descriptions or titles. The cleaning pipeline also transforms data (timestamps, salary normalization, derived columns) — that's equally important even if few rows are dropped.

"For salary stats, we didn't account for 50%, 75%, max"
.describe() actually does show 25%, 50%, 75%, max by default. If it's not showing, it's likely because the output got truncated in VS Code. You can explicitly print them:

"Is saving as Parquet the standard/professional?"
Yes — Parquet is the industry standard for analytical data. Every data team at companies like Netflix, Spotify, Airbnb uses Parquet (or the similar format Delta Lake/Iceberg). CSV is only used for raw ingestion. Reasons: 5x smaller, 10x faster to load, preserves column types (no more "is this a string or an int?" problems).

"How to interpret the boxplot (Salary by Experience Level)?"

The box = middle 50% of salaries (25th to 75th percentile)
The line inside the box = median salary (50th percentile)
The whiskers (lines extending from the box) = range of typical values (1.5x the box width)
Dots beyond the whiskers = outliers (unusually high/low salaries)
What to look for: Does the median increase from Entry → Director? Is the spread (box width) larger for some levels? Are Executive salaries much higher? This directly answers Theme 1.
"Days open stats: all the other stats are 0?"
This means most postings have days_open = 0 — they were opened and closed on the same day (or the timestamps are the same). This is actually an important finding for Theme 2 (Ghost Jobs): many postings close immediately, while a long tail stays open for weeks/months. The interesting ghost job candidates are in that long tail. We might want to filter to days_open > 0 for the ghost job analysis.

"How to interpret the heatmap?"

Each cell shows the Pearson correlation between two variables (-1 to +1)
+1.0 (dark red) = perfect positive correlation (when one goes up, the other goes up)
-1.0 (dark blue) = perfect negative correlation (when one goes up, the other goes down)
0.0 (white) = no linear relationship
What to look for: min/med/max salary should be highly correlated with each other (~0.9+). Views and applies should be positively correlated. days_open might be negatively correlated with applies (ghost jobs get fewer applications). The diagonal is always 1.0 (a variable correlates perfectly with itself).
